# SAT Coding Standards

Standards for code generated by SAT and for SAT's own codebase.
Injected into agent prompts during code-generating phases.

## Python Style

- **Python 3.12+** target
- **120-character line length** (configured in ruff.toml)
- **Double quotes** for strings
- **Spaces** for indentation (4 spaces)
- **LF** line endings (enforced by .gitattributes)
- **isort** for import ordering (stdlib → third-party → local)

## Code Organization

- One class per file for substantial classes
- Related small classes/functions may share a file
- Tests mirror source structure: `src/agents/foo.py` → `tests/test_foo.py`
- No circular imports — use dependency injection or late imports

## Error Handling

- Never use bare `except:` — always catch specific exceptions
- Use `raise NewError(...) from original_error` to preserve stack traces
- Transient errors (network, rate limit, timeout) → retry with backoff
- Persistent errors (import, syntax, logic) → fail fast, log, escalate
- Resilience wrappers: failures in non-critical paths (vector memory, notifications) must not crash the main workflow

## Data Validation

- Use **Pydantic models** at system boundaries (LLM output, config files, API responses)
- Internal data may use dicts/TypedDicts for performance
- Validate early, fail with clear error messages

## Testing

- **TDD**: Write tests before implementation code
- **pytest** as test framework
- **unittest.mock.patch** for external dependencies (LLM calls, subprocess, file I/O, network)
- Every test must be independently runnable — no test ordering dependencies
- Integration tests marked with `@pytest.mark.integration`
- Minimum test coverage: one test per public method, edge cases for critical paths

## Security

- No credentials in source code — use environment variables or config files outside repo
- No `eval()` or `exec()` on user input or LLM output
- Subprocess calls: avoid `shell=True` when possible; when required, validate inputs
- SQL: use parameterized queries (no f-strings in SQL)
- File paths: validate/sanitize paths from external input before use

## LLM Integration

- All LLM calls go through the routing engine — never hardcode a model
- LLM output must be parsed as JSON with Pydantic validation
- Auto-retry once on validation failure (with error feedback in prompt)
- Log all LLM calls: model, tokens, latency, cost
- Artifacts flow between phases; reasoning/thinking stays local

## Git Practices

- Commit messages: `type(scope): description` (conventional commits)
- Types: feat, fix, chore, ci, docs, test, refactor
- All code changes on branches, merged via PR
- Pre-commit hooks must pass before commit
- No force pushes to master

## What NOT To Do

- Do not add docstrings to obvious methods (getters, setters, simple wrappers)
- Do not add type annotations to test code unless it improves clarity
- Do not create abstractions for single-use patterns
- Do not use language-specific examples in templates that don't match the project's tech stack
- Do not over-engineer: three similar lines are better than a premature abstraction
